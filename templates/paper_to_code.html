What the paper uses and how we adapted it for product recommendation

Step 1 — Building the R Matrix

Paper: explicit user-item ratings only  →  Code: explicit + implicit behavior

📄 Paper — Section II
R is a user × item matrix where R[u][i] = explicit rating (1–5).

Entries not in the dataset are considered unknown (missing).

The paper uses MovieLens ratings.csv directly, no preprocessing needed. All ratings are already clean integers 1–5.
💻 Our Code — build_R()
def build_R(ratings_raw, behavior_raw):
  R = {}
  # Priority 1: explicit ratings
  for row in ratings_raw:
    R[(uid, pid)] = rate

  # Priority 2: implicit behavior
  score = 1.0 + v*0.1 + c*0.5 + p*2.0
Why we added implicit behavior: Real product datasets often have sparse explicit ratings. Using behavioral signals (views, clicks, purchases) lets us infer user interest even when a user never explicitly rated a product. This enriches R without changing the GA structure at all.
Implicit Score Formula (our addition):
Implicit Rating from Behavior
score = min(5.0, 1.0 + viewed×0.1 + clicked×0.5 + purchased×2.0)
A user who viewed (weak signal), clicked (medium), and purchased (strong) gets a high implicit score. Capped at 5.0 to match the explicit rating scale. Explicit ratings always take priority over implicit ones.

Step 2 — Building the P Matrix (Pseudo Tag Genome)

Paper: 1128 crowd-sourced tag scores  →  Code: 9 engineered feature dimensions

This is the most significant adaptation in our implementation. The paper has a ready-made MovieLens tag genome file. We had to build an equivalent from scratch using only product metadata and behavioral signals.

📄 Paper — Tag Genome
The MovieLens "genome" file contains 1128 tags like:

• "suspenseful" → Movie A: 0.85, Movie B: 0.12
• "visually stunning" → Movie A: 0.91, Movie B: 0.44
• "based on novel" → Movie A: 0.05, Movie B: 0.78

These are human-assigned relevance scores in [0,1] for every (tag, movie) pair. They are pre-computed and given — the paper does not build them.
💻 Our Code — build_P()
# Category one-hot (N_cat dimensions)
cat_vec = [1.0 if c==cat else 0.0]

# Normalized price
price_f = price / max_price

# Behavioral popularity
view_f = views[pid] / max_views
click_f = clicks[pid] / max_clicks
purch_f = purch[pid] / max_purchases

P[pid] = cat_vec + [price_f, view_f, click_f, purch_f]
Why these 4 types of features? — Mapping to tag genome semantics:
Tag Genome Concept → Our Feature Equivalent
"Is this item in genre X?" (e.g., genre: Action, Comedy) → Category one-hot: 1.0 if product is in category C, else 0.0
"Is this item expensive?" (luxury/budget appeal) → Price level: product_price / max_price_in_dataset
"Is this item popular / trending?" → View-count: total views / max views across all products
"Does this item drive engagement?" → Click-rate + Purchase-rate: normalized clicks and purchases
Key constraint maintained: All P values must be in [0,1] — exactly as in the paper. This ensures Q·Pᵀ also lands in [0,1], maintaining mathematical consistency. We achieve this by dividing every feature by its maximum value across the dataset.

Step 3 — GA Initialization

Paper: Equation (3) → Code: init_population()

Paper — Equation (3): Population Initialization
Q_k = {q_ui} where q_ui ∈ Uniform[0, 1]
Each chromosome k is a Q matrix. Every element q_ui (user u's affinity for tag i) is drawn from a uniform random distribution between 0 and 1.
📄 Paper Pseudocode (Algorithm 1, line 1)
FOR k = 1 to Np:
  Initialize Q_k with random values in [0,1]
  (each row = one user's tag affinity vector)
💻 Code — init_population(Np, n_users, n_tags)
return [
  [[random() for _ in range(n_tags)]
   for _ in range(n_users)]
  for _ in range(Np)
]
Perfect 1:1 mapping. The code is a direct 3-level nested list comprehension: outer level = Np chromosomes, middle level = n_users rows, inner level = n_tags genes. Each gene = random.random() which returns a float in [0.0, 1.0].
🎯

Step 4 — Fitness Function (RMSE)

Paper: Equation (4) → Code: compute_rmse() with scale fix

Paper — Equation (4): Fitness Function
F(Q_k) = RMSE(R, R̂_k) = √( (1/|Ω|) × Σ (r_ui − r̂_ui)² )
Ω = set of known (user, item) pairs with actual ratings.
r_ui = actual rating, r̂_ui = predicted rating = Q[u] · P[i].
❌ The Scale Problem
In the paper, R ∈ [1,5] and Q·Pᵀ also reaches [1,5] because there are 1128 tags — the cumulative dot product naturally covers [1,5].

In our system, we have only 9 tag dimensions. Q·Pᵀ is bounded in [0, 9×1 = 9], but in practice stays in [0, 1] because most category one-hot entries are 0.

Computing RMSE between R∈[1,5] and Q·Pᵀ∈[0,1] gives a meaningless, inflated error.
✔ Our Solution: Normalize R to [0,1]
Before computing RMSE, we normalize actual ratings to [0,1]:

actual_norm = (actual_rating − 1) / 4

This maps: 1 → 0.0, 3 → 0.5, 5 → 1.0

Now both sides are in [0,1], and RMSE is meaningful. The GA still minimizes the same error — just on a different scale.
def compute_rmse(Q, P, R, user_ids):
  uid_idx = {uid: i for i, uid in enumerate(user_ids)}
  sse, n = 0.0, 0

  for (uid, pid), actual in R.items():
    actual_norm = (actual - 1.0) / 4.0 # normalize R to [0,1] — our adaptation
    predicted = dot(Q[uid_idx[uid]], P[pid]) # = Q[u] · P[i]
    sse += (actual_norm - predicted) ** 2
    n += 1

  return math.sqrt(sse / n) if n > 0 else float('inf')

Step 5 — Roulette Wheel Selection (RWPSA)

Paper: Section III-A, RWPSA → Code: roulette_select()

Paper — Section III-A: Roulette Wheel Parent Selection Algorithm (RWPSA)
P(select Q_k) = (1/F_k) / Σ(1/F_j) where F = RMSE
Selection probability is inversely proportional to RMSE. Chromosomes with lower fitness (RMSE) have a higher probability of being selected as parents.
📄 Paper Description
1. Compute weight w_k = 1/F_k for each chromosome
2. Sum all weights: W_total = Σ w_k
3. Generate random r ∈ [0, W_total]
4. Walk through chromosomes accumulating weights
5. Select the chromosome where cumulative weight ≥ r
💻 Code — roulette_select()
weights = [1.0 / (f + 1e-10) for f in fitnesses]
total = sum(weights)
pick = random() * total
cumul = 0.0
for chrom, w in zip(population, weights):
  cumul += w
  if cumul >= pick:
    return chrom
The 1e-10 guard: We add a tiny epsilon to the denominator to prevent division-by-zero if a chromosome accidentally gets RMSE = 0 (perfect fit). This is a standard numerical safety guard not explicitly mentioned in the paper.
✂

Step 6 — Row Crossover

Paper: Section III-B, Example 2 → Code: crossover()

Paper — Section III-B: Single Row Crossover
Select random row index r ∈ [1, n_users]
Child 1 = rows of Parent 1, except row r which comes from Parent 2.
Child 2 = rows of Parent 2, except row r which comes from Parent 1.
Applied with probability Pc. If not applied, children = copies of parents.
📄 Paper Example 2 (paraphrased)
Parent A = [Row1_A, Row2_A, Row3_A]
Parent B = [Row1_B, Row2_B, Row3_B]
Crossover point = row 2

Child 1 = [Row1_A, Row2_B, Row3_A]
Child 2 = [Row1_B, Row2_A, Row3_B]
💻 Code — crossover(par1, par2, Pc)
if random() > Pc:
  return par1.copy(), par2.copy()

idx = randint(0, len(par1)-1)
child1 = [par2[i] if i==idx else par1[i]
          for i in range(len(par1))]
child2 = [par1[i] if i==idx else par2[i]
          for i in range(len(par2))]
Direct translation. The code is a direct Python implementation of the paper's description. The list comprehension with the conditional par2[i] if i==idx else par1[i] precisely replicates the row-swap logic from Example 2 of Section III-B.

Step 7 — Gene Mutation

Paper: Section III-C, Example 3 → Code: mutate() with adjusted Pm

Paper — Section III-C: Gene Mutation
q_ui_new = Uniform[0,1] if rand() < Pm, else q_ui_new = q_ui
For each gene in the chromosome: with probability Pm, replace it with a new random value from [0,1]. Otherwise, keep the gene unchanged.
📄 Paper Default: Pm = 0.5
With 1128 tags, a 50% mutation rate means about 564 genes mutate per user row. The large tag space means the GA needs aggressive mutation to explore the high-dimensional search space.
💻 Our Default: Pm = 0.05
# Only 9 tag dimensions → gentler mutation
# Pm=0.5 would randomize ~4-5 genes per row
# — too disruptive for only 9 genes
return [[random() if random() < Pm
         else g for g in row]
        for row in chrom]
Why we reduced Pm: The paper's Pm=0.5 is calibrated for 1128 dimensions. With only 9 tags, Pm=0.5 would randomize ~4–5 of the 9 genes per user row — essentially destroying most of the parent's information each generation. We use Pm=0.05 (1 gene in 20 on average), but the user can experiment with higher values in the demo.

def run_ga(P, R, user_ids, Np, Ni, Pc, Pm, seed):
  # Step 1: Initialize (paper Algorithm 1, line 1)
  population = init_population(Np, n_users, n_tags)
  fitnesses = [compute_rmse(ch, P, R, user_ids) for ch in population]

  # Track global best for elitism (paper Algorithm 1, line 2)
  best_Q = population[argmin(fitnesses)]

  for _ in range(Ni):
    next_pop = [best_Q.copy()] # elitism: carry best forward always
    next_fit = [best_rmse]

    while len(next_pop) < Np:
      p1 = roulette_select(population, fitnesses)
      p2 = roulette_select(population, fitnesses)
      c1, c2 = crossover(p1, p2, Pc)
      c1 = mutate(c1, Pm); c2 = mutate(c2, Pm)
      # Add children to next generation
      next_pop.append(child); next_fit.append(f)

    population = next_pop; fitnesses = next_fit
    # Update global best if this generation improved
    if fitnesses[gen_best_i] < best_rmse:
      best_Q = population[gen_best_i]

  return best_Q, history
Elitism implementation detail: The paper says "preserve the best chromosome". In code, we start each new generation's population with [best_Q.copy()] — the best chromosome seen across ALL previous generations (not just the current generation). Then we fill the rest via selection/crossover/mutation.

Step 9 — Generating Recommendations

Paper: R̂ = Q × Pᵀ → Code: recommend() with top-N filtering

Paper — Core Prediction Formula
r̂_ui = Q[u] · P[i] = Σ_t (q_ut × p_it)
The predicted rating of user u for item i is the dot product of their Q row with item i's P vector. This sums up: for each tag t, (how much user u likes t) × (how much item i relates to t).
📄 Paper Output
The paper outputs R̂ (full predicted matrix) and evaluates overall RMSE.

Recommendation = find top-N items with highest predicted rating for each user that are NOT in their known rating set.
💻 Code — recommend()
q_row = Q[uid_idx[uid]]
already = {pid for (u,pid) in R if u==uid}

scores = []
for pid, p_vec in P.items():
  if pid in already: continue
  scores.append((pid, dot(q_row, p_vec)))

scores.sort(key=lambda x: -x[1])
return scores[:top_n]
The dot() helper in our code computes exactly Q[u]·P[i]: sum(x * y for x, y in zip(a, b)) — a pure Python dot product without any libraries.

Design Decisions Summary

Every deviation from the paper and why it was made

Paper Approach → Our Decision & Reason
1128 tag genome (MovieLens) → 9 engineered features — No tag genome available; we build one from product features. Maintains the same mathematical role (P matrix in [0,1]).
Explicit ratings only in R → Explicit + implicit behavior — Implicit signals (views, clicks, purchases) enrich sparse datasets. Explicit ratings take priority.
RMSE on R∈[1,5] scale directly → Normalize R to [0,1] before RMSE — Required because Q·Pᵀ with 9 tags cannot reach [1,5]. Formula: (r−1)/4.
Pm = 0.5 (50% mutation rate) → Pm = 0.05 default — With only 9 genes per row, Pm=0.5 is too destructive. User can increase in the demo.
Np=200–400, Ni=2000–4000 (MovieLens scale) → Np=20, Ni=80 defaults — Smaller dataset converges faster. User-configurable up to Np=200, Ni=500.
Sample data: MovieLens 20M dataset → Built-in sample: 10 users, 20 products — Works out of the box without any data files via build_sample_data().
